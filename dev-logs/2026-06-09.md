## 补充（同日）：平板/PC 响应缓慢修复 v2

- [x] 新增 `@Trace dataVersion` 计数器保障响应式链路
  - 根因分析：平板布局使用条件渲染 `if/else`，与手机的 `Tabs` 容器不同。ArkUI V2 中，条件渲染下的 `@ComponentV2` 通过 `@Param` 访问 `@ObservedV2` 对象的 `@Trace` 属性时，响应式依赖可能不稳固
  - 修复：在 `ClipboardViewModel` 中新增 `@Trace dataVersion: number`，每次 `applyFilters()` 和 `deleteAll()` 后递增；在 `ClipboardPage.build()` 开头用 `void(this.vm.dataVersion)` 显式注册依赖
  - 文件：ClipboardViewModel.ets、ClipboardPage.ets

- [x] 平板/PC 网格布局重构：嵌套 ForEach → Flex + FlexWrap.Wrap
  - 根因：旧布局用 `Row > ForEach(distributeToColumns()) > Column > ForEach(clips)` 嵌套 ForEach。当 item 在列间迁移（如置顶后从第 2 列跳到第 1 列），ArkUI V2 的 ForEach diff 不稳定，导致 UI 不刷新或极其缓慢
  - 修复：改为 `Flex({ wrap: FlexWrap.Wrap }) > ForEach(clips)` 单层 ForEach，每张卡片用 `width: '50%'` / `'33.33%'` 百分比宽度，Flex 自动换行处理多列。消除嵌套 ForEach 和 `distributeToColumns()` 方法
  - 优势：与手机端已验证的单层 ForEach 模式一致，响应式链路可靠
  - 文件：ClipboardPage.ets

- [x] 移除废弃的 `distributeToColumns()` 方法，替换为 `getCardWidth()`

## 技术要点

- `Flex + FlexWrap.Wrap` 是替代手动列分配的更可靠方案，单层 ForEach 避免了 ArkUI V2 嵌套 ForEach diff 不稳定的问题
- `@Trace dataVersion` 作为原始类型属性，比数组 `@Trace` 更可靠地触发响应式更新
- 条件渲染 (`if/else`) 与 `Tabs` 容器在响应式传播上存在差异，需额外保障

## 补充 v5（同日）：修复双击 + 卡片放大

- [x] **双击修复**：`onClick` + 独立 `TapGesture({count:2})` → `GestureGroup(GestureMode.Exclusive)`
  - 根因：ArkUI 中 `onClick` 立即响应对单击，不给双击手势留识别时间，`TapGesture({count:2})` 永远等不到第二下
  - 修复：两个 TapGesture(count:1 和 count:2) 放入 Exclusive 互斥组。双击时不会触发单击，单击立即响应
  - 文件：ClipCard.ets

- [x] **卡片放大**（平板/PC 专属，手机不变）

  - ClipboardPage md/lg 分支传入 `wideLayout: true`，SM 分支不变
  - 文件：ClipCard.ets、ClipboardPage.ets
